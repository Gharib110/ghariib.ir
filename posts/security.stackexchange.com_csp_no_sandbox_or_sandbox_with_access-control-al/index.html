CSP: no sandbox, or sandbox with Access-Control-Allow-Origin: "null"?

Published on: 2024-12-30T10:16:27Z

Summary: I host two public sites with WordPress, using different themes (but official ones, named "twenty-something") on an Apache server (Debian Bookworm), and Im updating my CSP. If I enable sandbox, I get a bunch of (silently) failed requests for (local I guess) fonts files due to "CORS Missing Allow Origin", because the origin of those requests becomes null due to the sandboxing. To prevent them, I can either disable sandbox, or enable it and add the header Access-Control-Allow-Origin: "null". I also tried sandbox allow-same-origin but of course it doesnt work, since the origin is null and not my own sites. Note that WordPress is not the only web application to behave like this, I had similar problems with Nagios and phpMyadmin, but since theyre on a host in a private network, with other means of access control in place, I just removed sandbox from the CSP. With my public sites, Id prefer to be more careful. As a sysadmin, Im not that versed in web security, and I fail to see the implications of allowing origin null for all requests in terms of XSS attacks, but I somehow sense that those implications could go far beyond the initial goal of just avoiding a bunch of failed requests. So, my question is, which one is the most secure (or the less insecure), a CSP without sandbox, or one with sandbox and the header Access-Control-Allow-Origin: "null" ?
