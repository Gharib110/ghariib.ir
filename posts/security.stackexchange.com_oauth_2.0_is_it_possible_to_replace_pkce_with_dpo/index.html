OAuth 2.0: Is it possible to replace PKCE with DPoP-like proof-of-possession?

Published on: 2024-12-31T03:38:20Z

Summary: So Im currently learning about Demonstrating Proof-of-Possession (DPoP) in Oauth after previously learnt about Proof Key for Code Exchange (PKCE). one interesting idea ive been thinking is, is it possible to replace the challenge/verifier keypair in PKCE with a challenge/signature strategy (possibily with the same keypair as the one used in DPoP)? so in PKCE we usually does it like: client generate random verifier, hash with sha256 into a challenge code, send it along Oauth inital request after resource provider redirect back to client along with authorization code, send the auth code along with verifier code to exchange with access token. if we go by the above idea, it would be like: client generate public-private keypair for DPoP. send public key along with OAuth initial request after resource provider redirect back to client along with authorization code, send the auth code along with random string+signature to exchange with access token. or event the whole request could be encapsulated inside JWT and signed with the previous keypair. I know this is more about protocol design instead of implementation details, but I think this could possibly simplify the flow as we only need to focus on 1 type of cryptographic operations instead several (asymmetric signature + hashing). so is this flow sensible? is there any particular issue asides from deviating from OAuth standards?
